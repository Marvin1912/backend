PR #145 (issue #136, merged onto master ~2026-06) added `nutrition.day_target_snapshot` (migration V1_7) so historical day summaries reflect the targets that applied on that day. Entity `DayTargetSnapshotEntity` uses the natural `entry_date` (LocalDate) as `@Id`, extends `BasicEntity` (CreationTimestamp/UpdateTimestamp), repository keyed by LocalDate.

**Why:** Without a snapshot, `MealEntryService.getDay(pastDate)` showed *today's* targets for past days. A snapshot is written on the FIRST meal entry for a date (`ensureDayTargetSnapshot`), and `getDay` prefers it, falling back to live `getTargets()` when absent.

**How to apply:** When reviewing snapshot writes, remember the assigned non-null `@Id` + no `@Version` + not `Persistable` means `save()` routes through Hibernate `merge()` (SELECT-then-UPDATE) — same mechanics as [[nutrition-profile-singleton]]. So the check-then-act in `ensureDayTargetSnapshot` (findById then save, no `@Transactional`) does NOT fail on a concurrent duplicate; the second writer silently overwrites. Harmless under the single-user assumption (deterministic identical targets) but flag it. RESOLVED by PR #161 (issue #152, branch fix/nutrition-get-day-targets-fallback): `getDay`'s no-snapshot fallback now calls `getTargets(date)` (date-aware) instead of live `getTargets()`, so un-snapshotted historical days reflect the targets that applied on that date. New test `getDay_NoSnapshot_HistoricalDate_UsesDateAwareTargets` pins it. The `getDays` range-view gap (still used no-arg `getTargets()` range-wide) was RESOLVED by PR #170 (issue #166, branch fix/nutrition-getdays-target-mismatch): `getDays` now resolves each snapshot-less day via `nutritionTargetService.getTargetsSync(date)` inside a `Mono.fromCallable(...).subscribeOn(boundedElastic())`-wrapped loop, catching `TargetCalculationException` per-date to null out only that day. Trade-off (flag in future reviews): `getTargetsSync(date)` does 2 queries/day (profile findById + weight findTopByEntryDateLessThanEqual...), so a snapshot-less N-day range is now ~2N queries — the `getDays` Javadoc still wrongly claims "cost does not grow with days." `getDay`/`getDays` are now symmetric (snapshot-first, date-aware fallback).

PR #158 (issue #147, branch feature/nutrition-transactional-meal-entry) made `ensureSnapshot` `@Transactional(REQUIRES_NEW)` + `saveAndFlush` + catch `DataIntegrityViolationException` (treated as success) to make concurrent first-entry snapshot creation idempotent, and extracted meal-entry writes into `MealEntryWriteService` (`@Transactional` create/update/delete) called via the proxy from `MealEntryService`'s `Mono.fromCallable(...).subscribeOn(boundedElastic())`. KEY REVIEW POINT (raised here): the `DataIntegrityViolationException` catch is *largely unreachable* because the assigned-@Id / no-@Version / non-Persistable entity makes `saveAndFlush` route through Hibernate `merge()` (SELECT-then-INSERT/UPDATE), so a concurrent duplicate almost always lands as a silent UPDATE, not a unique-violation INSERT — the constraint only fires in the tiny window where both txns' merge-SELECTs miss. The catch is harmless defensive code, not the actual concurrency fix; the real safety net under single-user is the merge overwrite. To make the catch meaningful the entity would need to implement `Persistable#isNew()` (force `persist()`). Atomicity caveat: REQUIRES_NEW commits the snapshot in its OWN tx BEFORE the outer createEntry tx commits, so a later failure in the outer tx rolls back the meal entry but leaves the snapshot — and a failed snapshot no longer rolls back the entry. This is acceptable per issue #147 intent (don't let snapshot failure lose the entry) but is NOT true all-or-nothing atomicity; flag it.

PR #157 (issue #153, branch feature/nutrition-refresh-snapshot-on-weight) extracted the write logic into a dedicated `DayTargetSnapshotService` (ensureSnapshot + refreshSnapshotIfExists) and wired `WeightService.create/update` to refresh the snapshot when a same-day weight is logged (snapshot was frozen at first meal entry, before weight). Dep direction WeightService -> DayTargetSnapshotService -> NutritionTargetService is acyclic. `entryDate` column is `updatable=false` but refresh re-sets the same PK value so it's a no-op (no PK mutation). Note: refresh path has no `@Transactional`/explicit flush — relies on Hibernate FlushMode.AUTO flushing the just-saved weight before `getTargetsSync`'s query; OK for single-user. Per-PR gap to watch: tests assert delegation/refresh-call counts (mocked service) but no test pins that the merge updates the existing row in place rather than inserting a second — that is covered indirectly by entity mechanics, not a test.
